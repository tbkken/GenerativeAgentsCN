# 全书优化协作流程 v0.2

## 文档目标

本文件记录《从聊天机器人到虚拟小镇》书稿后续优化时的人机协作原则。后续每次优化书稿，都应先读取并遵守本流程；如果同时维护 `$review-book-chapters` skill，本文件与 skill 应保持同一套判断标准。

## 基本流程

1. 用户提出一个具体优化点。
2. Codex 先判断任务性质：是局部改稿、跨章节同步、源码深读、可运行实验、截图复现、还是提交发布。
3. 如果用户只是提出疑问或要求先看范围，先扫描相关章节、段落、概念、图表、命令和脚手架，列出影响位置，不直接修改正文。
4. 如果用户已经明确授权执行，例如“继续”“执行吧”“全权交给你”“按照经验处理”，直接改稿，但要把修改批次控制在可核查范围内。
5. 修改时按章节主线、运行顺序和读者理解断点推进，不做无关重写。
6. 修改完成后，运行可行的验证命令，检查 Markdown 围栏、图号、截图链接、脚手架输出、`git diff --check` 和章节小结。
7. 用户要求提交到 GitHub 时，先确认工作区范围，只提交本次相关文件；如果工作区混有无关改动，不静默 `git add -A`。
8. 提交后用 `git push` 推送到远端，并在回复中说明 commit、分支、验证结果。

## 写作技巧沉淀机制

- 当用户指出某个修改方向并明确认可后，Codex 需要把该判断抽象为可复用的文章优化技巧，沉淀到本流程或 `$review-book-chapters` skill 中。
- 技巧沉淀不只记录“这次改了什么”，还要记录“为什么这样改更好”：它解决的是读者理解断点、证据不足、过渡突兀、抽象泛泛、实验不可复现，还是章节主线不清。
- 沉淀后的技巧应保持可执行：后续编辑时能据此扫描全书、定位相似问题、生成修改方案，而不是只留下笼统口号。
- 如果经验来自某一章，例如第 14 章世界模型或第 15 章智能体初始化，要提炼成跨章节规则，而不是只记录一次性案例。

## 章节主线规则

- 每章必须有一个清楚的任务。源码章节尤其不能变成“看到哪里讲哪里”的代码旅游，要先说明这个机制解决什么问题，再按真实运行依赖展开。
- 章节开头直接进入问题现场，少写“本章会讲”“为什么这样安排”这类作者自述；结构信息交给小标题、表格和 Mermaid 图承载。
- 小节标题要直接，例如 `角色设定如何进入提示词 prompt`，不要写成表演式问题，例如 `为什么要先看角色设定`。
- 如果一章包含多个重机制，例如 Planning、Reacting、Dialogue，需要拆章或重构；不能让读者同时持有三套输入、prompt、输出、失败模式。
- 大幅重构后，必须同步修改开头问题、图号、图注、小结、参考资料和下一章过渡，避免正文结构已经变了，小结仍停留在旧稿。

## 信息设计规则

- 比较、分类、字段解释、失败诊断优先用表格；不要让读者从长段落里自己归纳差异。
- 机制改变系统能力时，开头用对照表说明“没有它会怎样，有了它以后怎样”。
- 多步骤机制用 Mermaid 图展示生命周期；复杂图要拆成几张小图，不把两条逻辑挤在一张图里。
- Mermaid 节点、图注和正文术语使用“中文 English”形式，例如 `世界地图 Maze`、`地图格子 Tile`、`提示词 prompt`、`断点 checkpoint`。
- 裸英文只保留在代码、路径、命令、字段名和方法名中；正文第一次出现专业词时必须给中文意思。
- 有限集合必须完整列出。25 个角色、固定 prompt 链、有限 provider、固定地图区域等，如果读者需要整体视野，就不要只抽样。

## 源码章节规则

- 源码章节按运行依赖顺序写，不按作者想到的顺序写。典型顺序是：真实数据文件 -> 配置结构 -> 管理对象 -> 单元对象 -> 状态或事件 -> 查询与更新 -> 主观记忆 -> 运行行为 -> 限制与排错。
- 不要只贴片段代码。每个关键源码段都要说明输入、分支、输出、状态变化和下游影响。
- 代码字段第一次出现时，用表格解释中文含义、来源文件、进入模块和行为影响。
- 配置驱动的概念必须展示真实配置。不要用 `{ ... }`、`[...]`、`map: { ... }`、`tiles: [...]` 作为唯一证据。
- 层级配置要同时展示 schema 和真实实例，例如 `world / sector / arena / game_object` 要说明字段怎么定义、真实值是什么、运行时如何补成完整地址。
- 首次解释重要对象，例如 Agent、角色 persona、记忆 memory、日程 schedule、事件 event、provider、world object 时，必须给出参与文件、类、字段、prompt、运行模块和一个真实填充案例。

## Prompt 规则

- 只要一个机制由大语言模型 LLM 驱动，就要在机制讲解处展示真实 prompt 路径、完整模板、变量含义、输出 schema 或 Pydantic 字段、输出流向。
- Prompt 不是附录材料。概念讲得再好，如果不知道真实 prompt 怎么写，读者仍然不知道如何驱动 Agent。
- 多个 prompt 形成链路时，先用表格列出阶段、prompt 文件或方法、输出和行为作用，再逐个展示关键模板。
- 如果书稿需要中英文对照，prompt 要给中文解释和英文含义；项目本身是中文 prompt 时，英文用于帮助读者理解意图，不替换真实模板。
- 增加 prompt 后必须检查 Markdown 围栏数量、参考资料路径和章节小结。

## 可运行证据规则

- 源码深读不能只靠片段解释，能跑的机制要配一个小脚手架。脚手架不必通用，但必须让当前小节的机制可见。
- 脚手架应放在稳定路径，例如 `docs/book/scaffolds/part_03/`，并提供从仓库根目录运行的命令。
- 书稿要贴真实 stdout、生成文件或截图，并用表格把输出行映射回源码片段。
- 如果脚手架刻意避免 LLM/API 调用，要明说边界；如果需要配置 provider，也要说明使用的环境变量或配置位置，但不要把密钥写进正文、截图、日志或提交。
- 可视化、前端、Phaser、回放、地图、canvas、WebGL 必须给截图，并实际检查截图是否有黑块、空白画布、贴图丢失、字体错位、时间戳不一致等问题。
- 捕获回放页面时，透明图层可能露出渲染器黑底；优先设置页面或渲染背景、等待有意义帧、必要时强制 Canvas 渲染，并做黑色像素比例之类的简单 sanity check。

## 命令与结果规则

- 命令不能裸奔。出现命令时，要说明工作目录、输入文件、关键参数、调用脚本和输出产物。
- 教程型命令必须尽量真实执行，正文写真实结果，不写凭空预测结果。
- 命令结果如果是网页、回放或 UI，必须截图并解释界面区域看到什么、说明什么。
- 纯后端命令至少要给 stdout、JSON 片段、生成文件路径或统计结果。
- 本地环境需要 `.venv/bin/python`、`python3` 或特殊工作目录时，要用实际可运行命令，不强行写成理想命令。

## 语言风格规则

- 正文要像技术作者在直接讲事情，不像模型在解释自己为什么要解释。
- 删除“读者应该”“本章会带你”“这里我们要注意”这类编辑过程话术，改成事实、判断、命令、结果。
- 不要把十个字以下的短句单独排成一行造成碎碎念；相关上下文合成完整句，保留自然段落节奏。
- 可以用表格、图、代码块降低阅读压力，但不要用空泛插图替代真实结构。
- 章节小结不是术语清单，要说明本章解决了什么、读完后应该能判断什么，并自然衔接下一章。

## 已沉淀技巧

- 代表性原文 + 项目解读：讲解项目配置、运行结果、日志、JSON、Markdown 报告或实验输出时，优先贴一段真实项目原文，再结合字段、路径、运行目标和任务进行解读，避免只做泛泛说明。
- 解读到实操的承接句：从文件内容、证据读法或实验结论转入命令行操作、浏览器回放、扩展实验之前，必须补一到两句自然过渡，说明“为什么现在可以进入下一步”，避免解释段落和实操步骤之间突然跳转。
- 作者直接论断，避免编辑腔：正文中不要写“这里不能一笔带过”“为什么要这样写”这类面向编辑过程的说明。作者应直接表达判断、事实和结论，让读者看到观点本身，而不是看到写作策略。
- 不解释写作选择：章节正文中不要反复解释“本章为什么这样安排”“为什么不讲某内容”“这样写有什么好处”。作者直接给出分工、边界、结论和操作；需要承接时，用内容推进句，而不是写作说明句。
- 开场直接给动作入口：章节开头少做章节安排说明，优先交代本节要完成的实验、配置或判断，让读者快速进入问题现场；结构信息交给小标题、图表和任务表承载。
- 硬编码值必须交代来源：正文出现坐标、ID、阈值、路径、端口等不透明配置值时，必须说明它从哪个文件、哪段输出或哪条命令得到，并给出至少一种验证方法，避免读者只能照抄不能迁移。
- 源码章节必须闭环：进入源码深读后，每个关键小节都要贴代表性真实代码，并在代码旁边完成字段、分支、输入输出和运行效果解读；有界面效果的内容必须给截图或真实渲染图，纯后端逻辑至少给脚手架输出、JSON 片段或表格化结果，避免读者一边看书一边频繁跳回仓库补上下文。
- 命令行不能裸奔：正文出现可执行命令时，必须拆解命令入口、工作目录、输入文件、关键参数、执行路径和输出产物；如果命令调用脚本，还要摘出脚本中的关键路径和核心计算代码，让读者知道这条命令为什么会产生后面的结果。
- 注释型代码块优先：讲解脚本、函数或配置生成逻辑时，优先把说明写进代码注释，再用普通 Markdown 语言标签整段展示，例如 ` ```python` 或 ` ```json`，以保留语法高亮；不要用注释伪造行号。若当前出版/预览链路不支持自动行号，就直接说明不支持，正式排版时再由支持行号的渲染器统一生成。
- 总起图按代码链路分层：源码章节开头的总起图不能只画概念名词，应按真实文件、类、函数、输入输出和运行结果组织；用大框突出章节要讲的层次，让后续小节能逐一回到图中的节点解释。
- 第 15 章经验：角色初始化这种首次系统概念，必须完整展示参与文件、公共配置、角色配置、有限角色清单、基础 prompt、真实填充示例、初始化流程、运行模块表和脚手架输出；否则读者只看到 `agent.json` 字段，无法理解角色如何进入 Agent、Maze、Scratch、Spatial、Schedule 和 Associate。

## 验证与提交规则

- 修改后至少检查：Markdown 围栏是否成对、图号是否连续、引用路径是否存在、章节小结是否同步、`git diff --check` 是否通过。
- 涉及脚手架时必须重跑脚手架，并把当前真实输出写回正文。
- 涉及截图时必须打开图像做视觉检查，不只看文件是否生成。
- 涉及 API key、日志、截图、配置 diff 时必须检查是否泄露密钥。
- 提交前运行 `git status -sb` 和 `git diff --name-status`，确认只提交相关文件。
- 如果当前分支是用户长期使用的主分支，且用户明确说“提交到 GitHub”或“push 到 GitHub”，可以直接 commit 并 push；如果用户要求 PR，或仓库协作模式要求 PR，再开 `codex/...` 分支。

## 适用范围

本流程适用于：

- 全书结构优化。
- 单一写作原则的全书统一。
- 章节开头、结尾、过渡、小结统一。
- 术语、引用、图表、实验任务、代码路径一致性检查。
- 源码深读章节的可运行脚手架、真实配置、真实 prompt、截图和结果解释。
- 面向本科生、研究者、开发者等不同读者层次的可读性优化。
- 后续需要提交到 GitHub 的书稿修改。
